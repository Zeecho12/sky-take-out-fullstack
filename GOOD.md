# GOOD.md —— 教学项目 AI 改造方法论(可移植主文档)

> **这是一份可移植的通用工作流方法论,不绑定任何具体项目。**
> 用户用它来系统性地改造从 GitHub 下载的教学项目,目标是学习工业级全栈开发
> + 准备北美后端 SDE 面试。第一个应用对象是"苍穹外卖"(Java monolith),
> 但这套方法论要复用到以后**任何**下载的教学项目上。
>
> 保真度优先:本文件完整记录了方法论的**结论 + 背后的理由**,让读到它的 AI
> 不只知道"做什么",还知道"为什么这么做"。

---

## 0. 给读到这份文档的 AI —— 先读这一段

如果你在一个**新的、刚下载的教学项目**里读到这份文档,你的任务**不是立刻改代码**,而是:

1. **理解这套方法论**(读完全文)。
2. **为当前项目生成专属文档**:根据本文档 §9 的模板——
   - 生成 `CLAUDE.md`(常驻宪法:填入该项目技术栈、改造目标、铁律、文档地图);
   - 生成一份**薄** `docs/WORKFLOW.md`(项目运行速查卡;真实命令等 Phase 0 跑通后再填);
   - 建好 `docs/features/`、`docs/decisions/`、`docs/divedeep/` 目录。
   - ⚠️ **GOOD.md 本身不改、不分叉**——它是可移植主文档,方法论的唯一真相源。改进回流到主副本(见 §附)。
3. **和用户确认后,进入 Phase 0**(安全网:git + 跑起来 + 关键路径测试)。

**牢记两条最重要的原则**(贯穿全程):
- 用户是 **Tech Lead / 架构师**,不是被动验收员。**所有带设计决策的事,你必须先把
  各方案的优缺点摆出来 → 和用户探讨 trade-off → 用户拍板 → 你才实现**,并把决策
  写成 ADR。
- **外包劳动,自留决策**:读取量大 / 边界清晰的活可以派 subagent;需要和用户探讨
  的决策留在主窗口。

用户缺乏很多工程背景,遇到他可能不懂的概念,从"为什么需要它"讲起,别默认他懂。

## 1. 核心理念与角色定义

### 1.1 目标决定一切:学习,不是交付

这些项目的根本目的是**学习 + 面试**,不是交付一个能上线的产品。这个目标推翻了一个
常见的错误假设——"所有任务全部由 AI 完成,我只负责最终验收"。

**为什么"全自动化"和"学习"冲突**:如果 AI 把设计决策、重构、测试全做完,用户只在
最后点头,那他学到的约等于零。面试官问"你为什么选 JWT 不选 session""这个索引为什么
加在这列",他答不上来,项目再漂亮也是负分。

**解决办法**:把工作分成两类,分别对待(见 1.3)。

### 1.2 用户的角色:Tech Lead,不是验收员

用户的定位是 **Tech Lead / 架构师**:定方向、拆任务、审设计、做决策。AI 是执行团队。

> 类比(英雄联盟):AI 是打野,帮你清野、做视野、支援;但**什么时候打大龙、什么时候
> 开团,是你(指挥)的事**。不能把指挥权也交出去,否则这把赢了你也不知道怎么赢的。

### 1.3 分工:外包劳动,自留决策

| 类别 | 谁主导 | 例子 |
|---|---|---|
| **带设计决策**(要理解、要拍板) | **用户在环**,AI 给方案讲 trade-off | 选型、schema 设计、模块边界、API 契约 |
| **无决策含量的劳动** | AI 做,用户审 | 样板 CRUD、批量改名、补测试、写注释、格式化 |

### 1.4 决策模式:摆方案 → 探讨 → 拍板 → 写 ADR

用户缺乏很多技术背景,无法独立判断选型。所以决策流程固定为:

1. AI 把每个候选方案的**优缺点**摆出来(用户不懂的概念要讲清)。
2. 和用户**探讨 trade-off**。
3. 用户**拍板**。
4. AI **同时实现代码 + 把决策和理由写成 ADR**(见 §5),作为学习文档。

这样每做一个功能,用户就积累一份亲身经历过的面试素材,而不是背来的。

## 2. 改造前的关键判断(方法论层面的避坑)

### 2.1 "删除某客户端"常是伪命题 —— 先分清"有价值的业务后端"和"平台特定噪音"

教学项目常带有特定平台的客户端(如微信小程序)。用户的第一反应往往是"这个平台在北美
没用,删掉"。**但要警惕:客户端是前端,而服务这个客户端的后端业务逻辑,往往才是整个
项目里对 SDE 面试最有价值的部分。**

> 类比(餐厅):管理后台是"后厨管理系统"(排班、进货、报表);面向顾客的客户端后端
> 是"顾客点餐到上菜的全流程"(下单、支付、购物车、地址)。面试考的是后者。因为
> "客户端没用"就把后者一起删了,等于把最值钱的菜倒了。

**正确做法**:把"删除某客户端"重新表述为——**"保留并加固有价值的业务后端,只把
平台特定的集成(登录 / 支付 / 客户端本身)替换成目标市场的标准实现"**。

> 首个项目实例(苍穹外卖):不是"删微信小程序",而是——保留 `controller/user/*`
> 业务后端,把①微信登录换成 JWT / OAuth2、②微信支付换成 Stripe / mock、
> ③小程序客户端用 React/Web 替代。这才是能写进简历的项目。

### 2.2 改动有"深水区"意识 —— 所以必须先建安全网

看似简单的替换,常常牵一发动全身。例如"替换登录",如果原项目用第三方平台的用户
标识(如 openid)作为**全局用户身份**,那么替换登录 = 重构整个用户身份模型,所有
相关接口都要动。

**这不是"删一个文件",而是一次牵一发动全身的重构。** 正因如此,动手前必须先建
"安全网"(Phase 0)——否则改坏了都不知道。

## 3. 五阶段工作流(核心)

```
Phase 0 安全网 Safety Net   git + 跑起来 + 关键路径测试(动业务代码前的前提)
Phase 1 理解 Understand      扫描项目,产出架构总览(派 subagent 读海量文件)
Phase 2 规划 Plan            写 Requirement + 探讨写 ADR → 定契约 → 拆 Proposal(主窗口,用户在环)
Phase 3 执行 Execute         按 Proposal 一步一 subagent,串行卡测试门;契约定死后前后端可并行
Phase 4 验证收尾 Close-out   验证 + 合并回 main + 复核 ADR / 收口 divedeep backlog + 再生派生文档 + 更新快照
```

> 常见错误是只做中间三步(理解→规划→执行),**头尾各缺一块**:头缺 Phase 0(没有
> git+能跑+测试,"验收"就是空话),尾缺 Phase 4(功能做完就散了:没验证 / 没合并 /
> 派生文档漂移 / 学习资产没收口)。

### Phase 0 —— 安全网(Safety Net)

- **目的**:建立"能验收、能回滚、能验证"的基础。**动任何业务代码前必须完成。**
- **为什么**:如果策略是"AI 改、我验收",可没有 git 就看不了 diff、回不了滚;
  没有测试就不知道改完有没有弄坏别的功能;项目没跑起来就连"改之前什么样"都不知道。
- **步骤**:
  1. `git init` + 写根目录 `.gitignore`(排除构建产物、依赖目录、本地配置)+ 首次
     提交,锁定"已知良好基线(known-good baseline)"。
  2. 让项目真正跑起来(数据库、缓存、后端、前端都能起)。**跑通前不改任何业务代码。**
  3. 补关键路径测试(**characterization test / 特征测试** 思路:在不改代码的前提下,
     写测试把"当前行为"钉死记录下来,之后重构只要测试还绿就证明外部行为没变)。
     **够用即可,不追求覆盖率。**
- **谁做**:git / 调环境在主窗口(交互调试);写测试可派 subagent。
- **完成标准**:项目能跑;核心链路测试全绿;基线已 commit。
- **产出后**:把跑通用到的真实命令(build / run / test / 环境路径 / 本机踩坑)填进
  `docs/WORKFLOW.md`——这是 WORKFLOW.md 第一次被真正填内容的时机。

### Phase 1 —— 理解(Understand)

- **目的**:搞清架构、模块职责、耦合关系,建立"空间地图"。
- **步骤**:
  1. 扫描项目,产出 `docs/Backend_scan/`(Phase 1 专用文件夹):里面放 backend-scan 的
     **全部产出**——各步原始输出 `PROJECT_S1~SN_*.md` + 合并后的 `BACKEND_OVERVIEW.md`(架构总览,下游 AI 读它)。
     > 用户已有一套 `backend-scan-*` skill(单体版)专做这件事;若无,则
     > 由 AI 系统性地读目录结构、配置、启动类、调用链,产出等价的总览文档。
  2. 额外产出一张**耦合地图**:哪些模块互相依赖、哪些代码被多端共用。**它决定
     Phase 3 哪些任务能并行、哪些必须串行。**
- **谁做**:**派 subagent 做扫描**(读海量文件是最典型的外包场景,还能保护主窗口
  上下文),主窗口看报告。
- **完成标准**:能对着总览说清"每块是谁、负责什么、谁调谁"。

### Phase 2 —— 规划(Plan)

- **目的**:把"想改的地方"变成"可执行、可验收的文档",并把设计决策拍板留痕。
- **步骤**(按档位裁剪,见 §4.3 右-sizing):
  1. **写 Requirement**(需求文档):背景/目标/范围(In·Out)/**功能级验收标准**。
     规划期写一次、基线化。
  2. **探讨决策 → 写 ADR**:对每个"选 A 弃 B 且 B 是认真候选"的改动,走 1.4 流程。
  3. **定契约(contract-first)**:全栈功能先在 `docs/api-contract` 定死接口
     (路径 / 方法 / 请求 / 响应 / 状态码 / 错误格式)。**契约是前后端唯一事实来源;
     定死后前后端才能并行开工。**
  4. **拆 Proposal**(技术方案):现状 + 方案总览 + 会动的关键文件 + **实施清单
     (每步一个测试门 + 依赖标注)**。串行依赖的步骤不能丢给并行 agent。
  5. **计划评审关卡(双路必跑:内审 + 外审)**:草拟完 Requirement/ADR/Proposal 后,**同时**
     交两个独立评审者做敌对评审,融合意见、修订计划,再进 Phase 3——把"运动员≠裁判"从代码
     (Phase 3)延伸到计划(Phase 2 是唯一没有客观门的阶段:选型对错没有测试能判)。
     **不分级、不挑选:每份计划都过这两路。**
     - **内审(in-house)**:开一个**全新上下文**的敌对 subagent,任务是"驳倒方案"——漏掉的候选 /
       被美化的 trade-off / 没说出口的假设 / 过于乐观的验收标准 / 难回滚的单向门。
     - **外审(external)**:用 **DeepSeek**(异构外部模型)做同样的敌对评审(CLI 位置见项目
       WORKFLOW / 全局工具目录)。异构 = 不共享内审模型对方案的自我偏好。
     - **融合 → 修订**:把两路批评合并——**收敛项(两家都提)= 高置信,优先改;分歧项 = 各自补的
       覆盖,逐条判去留**。据此修订 Requirement/ADR/Proposal。
     - **回灌 ADR**:融合后的关键发现记成 Addendum / 决策行("评审提了什么、为何仍否 / 怎么改"),
       变成学习资产 + 面试料("你怎么验证你的设计")。
     > 用**结构化批判**、不用开放辩论(后者易被嗓门大的一方带偏、易附和塌缩)。ADR-0001 首次实战印证:
     > 两家收敛处置信最高、分歧处补覆盖——这正是双路并跑而非单审的价值。
  - 以上都落到 `docs/features/NNNN-slug/`(requirement.md + proposal.md;progress.md 到执行期才追加)。
- **谁做**:决策、融合与拍板**全程主窗口,用户在环**(绝不外包);内审派**全新上下文 subagent**、
  外审走 **DeepSeek CLI**——评审的"劳动"可外包,融合/拍板不外包。
- **完成标准**:Requirement 边界清晰;关键决策有 ADR;Proposal 每步有明确测试门;
  **计划已过内审 + 外审双路评审、两路意见已融合并回灌 ADR**。

### Phase 3 —— 执行(Execute)

- **目的**:按 Proposal 实施清单实现代码,每步过测试门再进下一步。
- **关键认知**:一个功能拆成"步骤 1→2→3→4、每步测试通过才进下一步"时,这是
  **顺序编排(接力赛),不是并行 agent team(团战)**。步骤有依赖,并行会翻车。
  > 类比:不是五人同时推三路,而是 solo 单带——推掉一塔、确认塔没了、再推二塔。
- **步骤**(单个功能):
  1. 开 `feature/<slug>` 分支。
  2. 逐步骤推进:派 subagent 实现 Proposal 步骤 N → 跑测试 → **用户审 diff + 确认
     测试过** → 把 Proposal 清单该步标 `TESTED` + 更新 Proposal 交接头 + **追加一条
     Progress 现场笔记** → 提交一个 commit → 进步骤 N+1。
  3. 契约定死的全栈功能,**前端实现与后端实现可并行**(各对契约写)。
- **两条铁规**:
  - **不让写代码的 subagent 自己宣布"测试通过"**(运动员兼裁判)。测试门由用户 /
    独立 verifier / 确定性脚本把守。**职责分离(separation of duties)。**
  - **编排权(何时推进下一步)留在主窗口**,不下放给 agent。
- **谁做**:编排 / 审查 / 把门在主窗口;每步实现可派 subagent。
- **完成标准**:Proposal 所有步骤 `TESTED`;功能能跑;准备验收。

### Phase 4 —— 验证与收尾(Verify & Close-out)

- **目的**:确认改动正确、合并回 main,并把本功能的收尾动作做全(含留好学习资产的入口)。
- **为什么不叫"学习"**:两类学习资产其实**不在本阶段产出**——ADR(广度)在 Phase 2 拍板、
  Phase 3 以 Addendum 追加;divedeep(深度)是用户手动、按需触发的**跨阶段**活动。所以本阶段
  真正的工作是一组**确定性收尾动作**;"学习"在这里是**收口**(复核 + 记入 backlog),不是产出。
- **步骤**:
  1. **验证**:关键路径测试全绿 + 应用能跑 + 用户逐个审 diff。
  2. **合并**:功能整体合并回 `main`。
  3. **复核学习资产 + divedeep 收口(强制检查点)**:看该功能 ADR 要不要补 Addendum、
     面试要点是否记全。**必做**:把本功能 ADR「面试要点」里所有 **中/高** 的 divedeep 候选
     摆出来(主题 + 含金量 + 入口 + 一句为什么),**主动问用户对哪几条触发 divedeep**。
     —— 候选的**收集**是跨阶段的(Phase 1/2/3 边走边攒进 ADR 面试要点);此处是**选择**检查点。
     `divedeep` 本身仍是**按需**:写哪几条由用户拍板(高=默认建议写、中=默认建议并入或缓写、
     低=不写),AI **不擅自生成**。
  4. **再生派生文档**:里程碑处**再生** `docs/Backend_scan/BACKEND_OVERVIEW.md`(及需要时的 S 系列)。
  5. **收尾快照**:覆盖式更新 `CLAUDE.md` 的「当前进度」。
- **谁做**:验证 / 合并 / 拍板在主窗口;独立复审、(按需)写 divedeep 可派 subagent。
- **完成标准**:功能已合并;DoD 全部满足;派生文档已再生;学习入口(ADR / backlog)已收口。

## 4. 文档架构

> **核心原则:按"生命周期 + 读者 + 更新纪律"分文档,不按主题分。**
> 每类文档的存在理由,都能兑换成一类面试火力(见 4.2 末列)。这套体系要达到:
> AI 每次对话前读相关文档不丢上下文 / 三个月后仍可追溯"当初为什么这么设计" /
> 换个 AI 读一遍就能接手 / 面试官问 trade-off 答得出。

### 4.1 三份常驻/半常驻文档:方法论只存一处

**为什么**:每开一个新窗口,AI 对之前一切一无所知(它"失忆")。跨窗口续接不靠 AI
重读全部代码,靠沉淀在 repo 里的文档。但这三份必须**各司其职、方法论绝不重复**:

| 文档 | 装什么 | 可移植? | 加载方式 |
|---|---|---|---|
| **GOOD.md** | **全部方法论**:5 阶段、文档体系、模板、协作模型、概念词典 | ✅ 项目无关,**唯一真相源** | 需要时读 / 新项目的种子 |
| **CLAUDE.md** | 项目宪法:身份、技术栈、改造目标、**当前进度**、铁律(摘要,细节指向 GOOD.md)、文档地图 | ❌ 项目特定 | **每窗口自动加载,必须短** |
| **docs/WORKFLOW.md** | **项目运行速查卡**:真实构建/启动/测试命令(带本机路径)、环境、本机踩坑、新窗口指路 | ❌ 项目特定 | 需要时查 |

> WORKFLOW.md 只写"在这台机器、这个项目上具体怎么敲命令",**方法论一律指向 GOOD.md,
> 绝不复述**。它不是方法论文档,是运行手册。
> 类比(餐厅):GOOD.md = 连锁《运营宝典》(每家店一样,可移植);CLAUDE.md = 这家店的
> 前台牌(我们是谁、今日状态、店规摘要);WORKFLOW.md = 贴在这间厨房墙上的操作便条
> (本店设备、燃气总闸在哪)。你不会把整本宝典抄到厨房墙上。

**贯穿铁则:自动加载的文档必须有界(bounded)。** CLAUDE.md 每窗口都加载,越短越好——
任何会随进度**无限增长**的内容(里程碑历史、完成流水)一律不进它;「当前进度」用**覆盖式
更新**保持固定长度,历史交给 `git log` 和各功能的 Progress。

### 4.2 一个功能的文档:五类 + 生命周期表

一个功能的工作文档拆成五类(外加"交接快照"这一段)。**判断"更新纪律"的准绳,是它
的生命周期,而不是它讲什么主题**:

| 文档 | 主要读者 | 生命周期 & 更新纪律 | 对应面试题 |
|---|---|---|---|
| **Requirement**(需求) | 你 + 冷启动 AI | **写一次**,规划期基线;范围若变,追一条"范围变更",不改原文 | "介绍个项目 / 解决什么问题 / 范围多大"(行为面) |
| **Proposal**(技术方案) | 你 + 干活的 subagent | **活文档**:实施清单就地翻状态;交接头覆盖 | "你会怎么设计 X / 怎么拆解这个改动" |
| **ADR**(架构决策) | 你(面试复习) | 集合层**追加**新 ADR;单条采纳后**不可变**,改主意写新 ADR supersede | "为什么选 A 不选 B"(经典 trade-off) |
| **divedeep**(源码深读) | 你(面试复习) | **追加**,每篇独立、写完基本不动 | "X 底层怎么工作 / 带我走一遍链路" |
| **Progress**(现场笔记) | 冷启动 AI + 3 个月后的你 | **追加**,只记 git 看不出的东西 | (间接)"说个你 debug 的经历"(STAR 素材) |
| *(交接快照 Handoff)* | 下一个窗口的 AI | Proposal 顶部一段,**覆盖式**,永远只写"现在" | — |

两点必须记牢:
- **交接快照不是独立文件**,是 Proposal 的头部。冷启动 AI 先读它拿快照,再读 Progress 拿历史。
- **DoD 不进 Requirement。** 完成定义(代码 + 测试 + 文档 + ADR 都更新才算 DONE)是
  **全站一套、每个功能都一样**,写在 CLAUDE.md 铁律。Requirement 里只写**功能级验收
  标准**(可观测、可测试、含负例)。这是 Acceptance Criteria(功能对不对)和 DoD
  (流程走没走完)的区别——别混成一坨。

### 4.3 七条贯穿设计原则(为什么这么切)

1. **AC ≠ DoD**:验收标准(功能行为,每功能不同)进 Requirement;DoD(流程门,全站一套)进 CLAUDE.md。
2. **高度切分**:Requirement 只写问题层 why(稳定、写一次);代码现状 / 涉及文件下移到
   Proposal(易变、归活文档)。
3. **ADR(为什么)vs Proposal(怎么做)**:Proposal 写方案形状 + 步骤,"为什么选它"
   只留指针 `见 ADR-NNNN`,**不重复论证**。否则两边会打架、漂移。
4. **ADR(广度)vs divedeep(深度)**:ADR 是"为什么 A 不 B"的卡片;机制怎么跑写进
   divedeep。**ADR 面试要点 = divedeep 的选题 backlog**,两者双向链接、形成闭环。
5. **测试门分家**:每步的测试门**标准**写在 Proposal(动手前定死);**实测结果**写在
   Progress(动手后记)。既让"每步一测试门"有牙齿,又守住"运动员 ≠ 裁判"。
6. **Progress 只记 git 看不出的**:验证证据 + 发现 / 踩坑 / 临场决策(**"发现/踩坑"是
   一等公民,必填,没有写"无"**)。别逐文件抄 diff——那交给 git。且 **Progress 记
   "事件",ADR Addendum 记"决策"**,决策只留指针,不重复。
7. **右-sizing 双开关**——别用"大小"这种主观标尺,只问两个 yes/no:
   - "要拆步骤吗?" → 要 → 有 **Proposal + Progress**;不要 → 进度靠 commit message。
   - "选型纠结吗(B 是认真候选)?" → 纠结 → 写 **ADR**;不纠结 → 不写。
   - 阶梯:**T0 微改**(typo/注释)无文档;**T1 小功能/bugfix**只写轻量 Requirement +
     commit;**T2 中等**(多步骤/跨模块)Requirement+Proposal+Progress、ADR 可选;
     **T3 大改造**(真 trade-off + 多步骤)全套 + 一或多份 ADR。

### 4.4 追溯脊柱 + ADR 纪律

- **追溯脊柱**:同一功能共用编号 `NNNN`;每份文件头放**交叉链接块**(兄弟互指 + 指
  ADR / 契约)。关系**非严格 1:1**——一个 Requirement 可对应 0..n 份 ADR、0..1 份 Proposal。
- **ADR 纪律**:一功能一份;顶部**决策概览表**(≥2 个决策必填,当导航);采纳后**不可变**。
  同功能执行期发现的细化 → 写 **Addendum**;后续功能推翻旧决策 → 开**新 ADR** 标
  `supersedes NNNN`,**绝不回改旧 ADR**(旧决策的"当时为什么"本身是学习资产)。

### 4.5 派生文档 vs 源头文档(治"文档漂移")

| | 派生文档(Derived) | 源头文档(Source-of-truth) |
|---|---|---|
| 例子 | 架构总览、Swagger API 文档 | Requirement、ADR、`api-contract`、Proposal、Progress |
| 本质 | 代码的"缓存",能从代码再生 | 含人类意图 / 决策,代码里生不出来 |
| 维护 | **不手动维护,里程碑再生** | **手动维护,列入 DoD** |

三条策略:①接口文档尽量用 **Swagger / 注解自动生成**(改注解即改文档,永不漂移);
②架构总览当"快照",每个 Phase / 大功能完成后**再生一次**;③把"更新文档"写进 **DoD**。

### 4.6 推荐目录结构(通用,C 混合布局)

```
<project-root>/
├─ <后端代码目录>/
├─ <前端代码目录>/
├─ reference/                        # [只读] 参考资料(被替换的旧客户端源码),禁改、不构建、不扫描
├─ CLAUDE.md                         # 常驻宪法,自动加载
├─ GOOD.md                           # [方法论] 可移植主文档(从主副本复制进来,不分叉)
└─ docs/
   ├─ WORKFLOW.md                    # [项目速查] 本机命令/环境/踩坑(方法论看 GOOD.md)
   ├─ Backend_scan/                  # [派生] Phase 1 backend-scan 产出,里程碑再生
   │    ├─ PROJECT_S1~SN_*.md        #   各步扫描原始输出
   │    └─ BACKEND_OVERVIEW.md       #   合并的架构总览(下游 AI 读它)
   ├─ api-contract/                  # [源头] 前后端接口契约
   ├─ features/                      # [源头/living] per-feature 工作文档,按功能共置
   │    └─ 0001-<slug>/
   │        ├─ requirement.md        #   写一次(基线)
   │        ├─ proposal.md           #   活文档:交接头 + 现状 + 方案 + 实施清单
   │        └─ progress.md           #   追加:现场笔记
   ├─ decisions/                     # [源头/永久] ADR,全局扁平(广度学习资产)
   │    └─ 0001-<slug>.md
   └─ divedeep/                      # [学习/永久] 源码精读,全局扁平(深度学习资产)
```

**为什么 features 共置、decisions/divedeep 全局扁平(C 混合)**:per-feature 文档共置 →
开一个文件夹看完一个功能的来龙去脉(利于冷启动 / 三个月后回看);ADR 和深读笔记全局扁平
→ 它们是**跨功能的面试题库**,你会想把所有技术决策一次摊开复习。**所有文档进 git、和
代码一起提交**——checkout 到历史提交时看到的是那时的文档状态。文件名用数字前缀 + slug
保证顺序可读(divedeep 例外:用描述性长标题,靠头部「关联」接入脊柱)。

## 5. 关键概念与协议(工程实践词典)

- **Requirement(需求文档)**:一个功能"做什么 + 为什么 + 范围 + 验收标准"。写一次、
  基线化。核心是**问题陈述 + 边界**,不写技术方案。对应面试的"介绍项目 / scoping"。

- **验收标准(Acceptance Criteria)vs 完成定义(Definition of Done, DoD)**:前者=
  "这个**功能**做对了吗"(可观测、含负例,每功能不同,进 Requirement);后者="我们的
  **流程**走完了吗"(代码+测试+文档+ADR,全站一套,进 CLAUDE.md 铁律)。别混。

- **Proposal(技术方案 / Design Doc)**:给定决策后,方案长什么样(组件 / 数据流 / 业务
  时序)+ 怎么分步实现(实施清单 + 依赖 + 每步测试门)。**"为什么选它"只指向 ADR,不
  重复论证**。对应面试的"你会怎么设计 X / 怎么拆解"。

- **ADR(Architecture Decision Record,架构决策记录)**:记录"我们当时为什么这么选"的
  标准格式,是本方法论的**核心学习资产(广度)**——面试最爱问"为什么用 X 不用 Y"。
  一功能一份、含决策概览表、采纳后不可变。模板见 9.5。

- **divedeep(源码深读笔记)**:某段源码"底层到底怎么跑"的精读(**深度**)。由 `divedeep`
  skill 生成,面向"合上项目、只读笔记即可复习"。ADR 管广度、divedeep 管深度,分工明确;
  ADR 的面试要点就是 divedeep 的选题清单。

- **Progress(现场笔记)**:执行期"每步发生了什么"的追加日志。**只记 git 看不出的**:
  验证证据 + 发现 / 踩坑 / 临场决策。不抄 diff。它约等于真实工程里的 **PR 描述**。

- **交接快照(Context Handoff)**:Proposal 顶部一段,覆盖式,只写"当前在哪 / 下一步 /
  别碰哪些文件 / 怎么验证"。新窗口读这一段 + 清单即可秒接。

- **契约优先(Contract-First)**:全栈功能实现前先定死前后端接口契约。它是前后端唯一
  事实来源,也是二者能并行的前提。

- **完成的定义(DoD)**:一个功能"完成"= 代码 + 测试 + 相关文档(requirement / proposal /
  progress)+ ADR 全部更新。文档更新是"完成"的一部分,不是事后补。

- **复述握手(Verification Handshake)**:新窗口 AI 读完状态后,**先向用户复述**"当前
  在哪步 / 下一步 / 要改哪些文件 / 怎么验证",用户确认后才动手。防止"理解偏了还狂奔"。

- **计划评审关卡(双路必跑:内审 + 外审)**:Phase 2 的独立怀疑者,把"运动员≠裁判"从代码延伸到
  计划。**为什么需要**:产出方案的模型对自己的选择有**自我偏好(self-preference bias)**、倾向
  为当初的推理辩护,而 Phase 2 是唯一没有客观门(测试)的阶段,单靠用户把关又受背景所限。
  **做法**:每份 Requirement/ADR/Proposal 都**同时**过两路——**内审**=全新上下文的敌对 subagent
  驳方案;**外审**=异构外部模型(DeepSeek)当 critic。**不分级、都必跑**,然后**融合**两路批评
  (收敛项高置信、分歧项补覆盖)再修订计划。用**结构化批判**而非开放辩论;评审产出回灌 ADR。详见
  §3 Phase 2 步骤5。

- **特征测试(Characterization Test)**:重构前先写测试把"当前行为"钉死(哪怕有 bug 也
  照实记录),之后改动只要测试还绿就证明外部行为没变。是"AI 改、我验收"的物质基础。

- **追溯脊柱**:同一功能共用 `NNNN` + 各文件头交叉链接块,保证"业务决策不悬空、进度
  可追溯"。

- **只读参考区(`reference/`)**:被替换的旧客户端 / 第三方参考代码放这里,标记只读、
  排除构建 / 扫描 / AI 修改。用它之前先榨取价值(如提取接口契约写进 `api-contract`)。

- **派生文档里程碑再生**:能从代码再生的文档(架构总览)不逐次手改,大节点重跑一次。

## 6. subagent 委派策略

**主窗口是用户的指挥席**:用户只跟它说话,它自己判断何时把活外包给 subagent。

判断规则:subagent 上下文**隔离**(看不到对话,只返回结果)——
- ✅ **外包**:边界清晰、读取量大、只要结果不要过程的活。
- ❌ **自留**:需要和用户反复探讨、用户必须在环做决策的活。

| Phase | 主窗口(用户在环) | 外包给 subagent |
|---|---|---|
| 0 安全网 | git、跑起来 | 写关键路径测试 |
| 1 理解 | 看报告、提问 | **整个扫描(读海量文件)** |
| 2 规划 | **写 Requirement、探讨写 ADR、定契约、拆 Proposal、融合评审、拍板** | **内审计划红队(全新上下文敌对 subagent)+ DeepSeek 外审(CLI)**;explore 查"某标识符在哪被用" |
| 3 执行 | 编排、审 diff、把测试门 | **每一步实现 = 一个 subagent** |
| 4 验证收尾 | 跑测试、审 diff、合并、拍板 | 独立 verifier 复审;(按需)`divedeep` 生成深读笔记 |

要点:
- **不是"一 phase 一 subagent"**:Phase 1 可能一个大 subagent;Phase 2 决策不外包、但计划
  内审红队派一个全新上下文敌对 subagent(外审走 DeepSeek CLI);Phase 3 是**一步一个**。
- **同一份 Proposal 一物两用**:对新窗口主 AI 是"交接文档",对派出的 subagent 是
  "任务简报"。所以 Proposal 要写清"现状 / 方案 / 要改哪些文件 / 每步测试门"。
- subagent 还能**保护主窗口上下文**:让它读几十个文件、只返回浓缩结果,主窗口始终清爽。
- **现阶段建议**:起步时能在主窗口做的就在主窗口做,只外包最明显的重活(扫描)。跑顺
  一个完整功能、有了手感,再逐步把 Phase 3 实现步骤也交给 subagent。

## 7. 分支与提交策略

- **一功能一分支**:`feature/<slug>`。
- **一步一 commit**:Proposal 实施清单里每个步骤对应一个 commit;message 说清"改了什么
  + 关联功能/步骤"。
- **一功能一次合并**:整个功能验收通过后合并回 `main`。
- 好处:审 diff 颗粒度细,任何一步出错都能单独回滚。

## 8. 元原则:先手动跑,后固化成 skill

方法论有双重身份:

- **现在**:手动跑流程的操作指南(就是 GOOD.md 本身各 Phase 段)。
- **将来**:某个 phase 跑顺、步骤稳定了,GOOD.md 里那节的内容**就是那个 skill 的规格
  说明**,可交给 skill-creator 固化成 skill(如 `backend-scan-*`、`divedeep` 就是这么来的)。

> 注意:**proto-skill 的规格在 GOOD.md(方法论),不在 WORKFLOW.md**(后者只是项目命令
> 速查,不是 skill 素材)。

**所以"先手动跑"不是浪费,是在给未来的 skill 攒真实规格。**
> 类比:先自己走一遍这条路线,记下"哪个路口容易走错",这份笔记才是将来做"自动导航
> (skill)"的数据。没走过就做导航,一定脱离实际。

**推荐节奏**:先纯手动跑通第一个真实功能(当小白鼠),跑顺了再把 phase 逐个固化成 skill。

### 9.7 divedeep(源码深读笔记)

- **不手写模板,由 `divedeep` skill 生成**;正文格式以 skill 的金标准样本为准。
- 唯一约定:头部有「关联」头——`关联: ADR-NNNN / feature NNNN`,纯理解型写
  `关联: 阅读理解 · 原项目 <模块>`,靠它接入追溯脊柱。
- 触发来源(三条):①**Phase 1 理解驱动**(读懂原项目某链路);②**Phase 2 ADR 面试要点**里
  评为「高」的 divedeep 候选;③**Phase 3 执行期**亲手追踪 / 写完某链路后当场识别出的高含金量机制。
  三处谁看到都随手记一条候选(主题 + 含金量 + 入口 file:method + 一句为什么),沉淀进**对应
  功能 ADR 的「面试要点」**(即 per-feature backlog,不另建文档)。
- **只有「高」默认触发**:高 = 默认进候选、Phase 4 主动问是否写;中 = 按需 / 并入高篇;低 = 不写。
  宁缺毋滥 —— 目标是每个功能**不漏掉那 1 条真正值钱的链路**,不是凑数量。

## 10. 用这份文档启动一个新项目(操作清单)

当用户下载了一个新的教学项目,想套用这套工作流时:

1. **把 GOOD.md 复制进新项目根目录**(从主副本复制,见 §附;GOOD.md 本身不改)。
2. **AI 通读 GOOD.md**,理解方法论。
3. **AI 为新项目生成专属文档**:
   - 据 9.1 生成 `CLAUDE.md`(填该项目技术栈、改造目标、平台特定点)。
   - 据 9.2 生成一份**薄** `docs/WORKFLOW.md`(命令等 Phase 0 跑通后再填)。
   - 建好 `docs/features/`、`docs/decisions/`、`docs/divedeep/`、`docs/api-contract/`。
4. **AI 先做 2.1 判断**:识别"有价值的业务后端"和"平台特定噪音",和用户一起定改造目标
   (先别删任何东西)。
5. **和用户确认后,进入 Phase 0**(安全网)。
6. 之后按 Phase 1→4 循环推进每个功能。

> 一句话:GOOD.md 是"种子",AI 读完它,为具体项目长出 CLAUDE.md + 薄 WORKFLOW.md +
> docs 结构,然后按统一工作流干活。**种子不变,各项目只长自己的枝叶。**
