# GOOD.md —— 教学项目 AI 改造方法论(可移植主文档)

> **这是一份可移植的通用工作流方法论,不绑定任何具体项目。**
> 用户用它来系统性地改造从 GitHub 下载的教学项目,目标是学习工业级全栈开发
> + 准备北美后端 SDE 面试。第一个应用对象是"苍穹外卖"(Java monolith),
> 但这套方法论要复用到以后**任何**下载的教学项目上。
>
> 保真度优先:本文件完整记录了方法论的**结论 + 背后的理由**,让读到它的 AI
> 不只知道"做什么",还知道"为什么这么做"。

---

## 0. 给读到这份文档的 AI —— 先读这一段

如果你在一个**新的、刚下载的教学项目**里读到这份文档,你的任务**不是立刻改代码**,而是:

1. **理解这套方法论**(读完全文)。
2. **为当前项目生成专属文档**:根据本文档第 9 节的模板,为这个项目生成
   `CLAUDE.md`(协作宪法)和 `docs/WORKFLOW.md`(操作手册),把通用方法论
   "实例化"到这个具体项目上。
3. **和用户确认后,进入 Phase 0**(安全网:git + 跑起来 + 关键路径测试)。

**牢记两条最重要的原则**(贯穿全程):
- 用户是 **Tech Lead / 架构师**,不是被动验收员。**所有带设计决策的事,你必须先把
  各方案的优缺点摆出来 → 和用户探讨 trade-off → 用户拍板 → 你才实现**,并把决策
  写成 ADR。
- **外包劳动,自留决策**:读取量大 / 边界清晰的活可以派 subagent;需要和用户探讨
  的决策留在主窗口。

用户缺乏很多工程背景,遇到他可能不懂的概念,从"为什么需要它"讲起,别默认他懂。

## 1. 核心理念与角色定义

### 1.1 目标决定一切:学习,不是交付

这些项目的根本目的是**学习 + 面试**,不是交付一个能上线的产品。这个目标推翻了一个
常见的错误假设——"所有任务全部由 AI 完成,我只负责最终验收"。

**为什么"全自动化"和"学习"冲突**:如果 AI 把设计决策、重构、测试全做完,用户只在
最后点头,那他学到的约等于零。面试官问"你为什么选 JWT 不选 session""这个索引为什么
加在这列",他答不上来,项目再漂亮也是负分。

**解决办法**:把工作分成两类,分别对待(见 1.3)。

### 1.2 用户的角色:Tech Lead,不是验收员

用户的定位是 **Tech Lead / 架构师**:定方向、拆任务、审设计、做决策。AI 是执行团队。

> 类比(英雄联盟):AI 是打野,帮你清野、做视野、支援;但**什么时候打大龙、什么时候
> 开团,是你(指挥)的事**。不能把指挥权也交出去,否则这把赢了你也不知道怎么赢的。

### 1.3 分工:外包劳动,自留决策

| 类别 | 谁主导 | 例子 |
|---|---|---|
| **带设计决策**(要理解、要拍板) | **用户在环**,AI 给方案讲 trade-off | 选型、schema 设计、模块边界、API 契约 |
| **无决策含量的劳动** | AI 做,用户审 | 样板 CRUD、批量改名、补测试、写注释、格式化 |

### 1.4 决策模式:摆方案 → 探讨 → 拍板 → 写 ADR

用户缺乏很多技术背景,无法独立判断选型。所以决策流程固定为:

1. AI 把每个候选方案的**优缺点**摆出来(用户不懂的概念要讲清)。
2. 和用户**探讨 trade-off**。
3. 用户**拍板**。
4. AI **同时实现代码 + 把决策和理由写成 ADR**(见第 5 节),作为学习文档。

这样每做一个功能,用户就积累一份亲身经历过的面试素材,而不是背来的。

## 2. 改造前的关键判断(方法论层面的避坑)

### 2.1 "删除某客户端"常是伪命题 —— 先分清"有价值的业务后端"和"平台特定噪音"

教学项目常带有特定平台的客户端(如微信小程序)。用户的第一反应往往是"这个平台在北美
没用,删掉"。**但要警惕:客户端是前端,而服务这个客户端的后端业务逻辑,往往才是整个
项目里对 SDE 面试最有价值的部分。**

> 类比(餐厅):管理后台是"后厨管理系统"(排班、进货、报表);面向顾客的客户端后端
> 是"顾客点餐到上菜的全流程"(下单、支付、购物车、地址)。面试考的是后者。因为
> "客户端没用"就把后者一起删了,等于把最值钱的菜倒了。

**正确做法**:把"删除某客户端"重新表述为——**"保留并加固有价值的业务后端,只把
平台特定的集成(登录 / 支付 / 客户端本身)替换成目标市场的标准实现"**。

> 首个项目实例(苍穹外卖):不是"删微信小程序",而是——保留 `controller/user/*`
> 业务后端,把①微信登录换成 JWT / OAuth2、②微信支付换成 Stripe / mock、
> ③小程序客户端用 React/Web 替代。这才是能写进简历的项目。

### 2.2 改动有"深水区"意识 —— 所以必须先建安全网

看似简单的替换,常常牵一发动全身。例如"替换登录",如果原项目用第三方平台的用户
标识(如 openid)作为**全局用户身份**,那么替换登录 = 重构整个用户身份模型,所有
相关接口都要动。

**这不是"删一个文件",而是一次牵一发动全身的重构。** 正因如此,动手前必须先建
"安全网"(Phase 0)——否则改坏了都不知道。

## 3. 五阶段工作流(核心)

```
Phase 0 安全网 Safety Net   git + 跑起来 + 关键路径测试(动业务代码前的前提)
Phase 1 理解 Understand      扫描项目,产出架构总览(派 subagent 读海量文件)
Phase 2 规划 Plan            探讨 trade-off → 写 ADR → 定契约 → 拆工单(主窗口,用户在环)
Phase 3 执行 Execute         一步一 subagent,串行卡测试门;契约定死后前后端可并行
Phase 4 验证学习 Verify      跑测试 + 审 diff + 写 ADR/面试笔记
```

> 常见错误是只做中间三步(理解→规划→执行),**头尾各缺一块**:头缺 Phase 0(没有
> git+能跑+测试,"验收"就是空话),尾缺 Phase 4(功能做完但没沉淀成面试能力)。

### Phase 0 —— 安全网(Safety Net)

- **目的**:建立"能验收、能回滚、能验证"的基础。**动任何业务代码前必须完成。**
- **为什么**:如果策略是"AI 改、我验收",可没有 git 就看不了 diff、回不了滚;
  没有测试就不知道改完有没有弄坏别的功能;项目没跑起来就连"改之前什么样"都不知道。
- **步骤**:
  1. `git init` + 写根目录 `.gitignore`(排除构建产物、依赖目录、本地配置)+ 首次
     提交,锁定"已知良好基线(known-good baseline)"。
  2. 让项目真正跑起来(数据库、缓存、后端、前端都能起)。**跑通前不改任何业务代码。**
  3. 补关键路径测试(**characterization test / 特征测试** 思路:在不改代码的前提下,
     写测试把"当前行为"钉死记录下来,之后重构只要测试还绿就证明外部行为没变)。
     **够用即可,不追求覆盖率**——只要项目能跑、改动的功能能正常发挥作用。
- **谁做**:git / 调环境在主窗口(交互调试);写测试可派 subagent。
- **完成标准**:项目能跑;核心链路测试全绿;基线已 commit。

### Phase 1 —— 理解(Understand)

- **目的**:搞清架构、模块职责、耦合关系,建立"空间地图"。
- **步骤**:
  1. 扫描项目,产出 `docs/BACKEND_OVERVIEW.md`(架构总览)。
     > 用户已有一套 `backend-scan-*` skill(单体版 / 微服务版)专做这件事;若无,则
     > 由 AI 系统性地读目录结构、配置、启动类、调用链,产出等价的总览文档。
  2. 额外产出一张**耦合地图**:哪些模块互相依赖、哪些代码被多端共用。**它决定
     Phase 3 哪些任务能并行、哪些必须串行。**
- **谁做**:**派 subagent 做扫描**(读海量文件是最典型的外包场景,还能保护主窗口
  上下文),主窗口看报告。
- **完成标准**:能对着总览说清"每块是谁、负责什么、谁调谁"。

### Phase 2 —— 规划(Plan)

- **目的**:把"想改的地方"变成"可执行、可验收的工单",并把设计决策拍板留痕。
- **步骤**:
  1. **探讨决策**:对每个带设计含量的改动,走 1.4 的决策流程,产出 ADR。
  2. **定契约(contract-first)**:全栈功能先在 `docs/api-contract.md` 定死接口
     (路径 / HTTP 方法 / 请求体 / 响应体 / 状态码 / 错误格式)。**契约是前后端的
     唯一事实来源;定死后前后端才能并行开工。**
     > 类比(餐厅):契约是"服务员和后厨之间的点菜单格式"。单子格式说定了,前厅
     > (前端)和后厨(后端)就能各自同时开工,不用互相等。
  3. **拆工单**:把功能拆成**有依赖顺序**的步骤,写进 `docs/changes/NNNN-*.md`
     (用第 9 节模板),标清范围边界、依赖、验收标准、学习点。**串行依赖的步骤不能
     丢给并行 agent。**
- **谁做**:**全程主窗口,用户在环**(这是决策,绝不外包)。
- **完成标准**:每步都有明确边界和验收标准;关键决策都有 ADR。

### Phase 3 —— 执行(Execute)

- **目的**:按工单实现代码,每步过测试门再进下一步。
- **关键认知**:一个功能拆成"步骤 1→2→3→4、每步测试通过才进下一步"时,这是
  **顺序编排(接力赛),不是并行 agent team(团战)**。步骤有依赖,并行会翻车。
  > 类比:不是五人同时推三路,而是 solo 单带——推掉一塔、确认塔没了、再推二塔。
- **步骤**(单个功能):
  1. 开 `feature/<slug>` 分支。
  2. 逐步骤推进:派 subagent 实现步骤 N → 跑测试 → **用户审 diff + 确认测试过**
     → 更新工单清单为 `TESTED` + 更新交接段 → 提交一个 commit → 进步骤 N+1。
  3. 契约定死的全栈功能,**前端实现与后端实现可并行**(各对契约写)。
- **两条铁规**:
  - **不让写代码的 subagent 自己宣布"测试通过"**(运动员兼裁判)。测试门由用户 /
    独立 verifier / 确定性脚本把守。**职责分离(separation of duties)。**
  - **编排权(何时推进下一步)留在主窗口**,不下放给 agent。
- **谁做**:编排 / 审查 / 把门在主窗口;每步实现可派 subagent。
- **完成标准**:所有步骤 `TESTED`;功能能跑;准备验收。

### Phase 4 —— 验证与学习(Verify & Learn)

- **目的**:确认改动正确,并把"AI 干的活"沉淀成"用户的面试能力"。
- **步骤**:
  1. **验证**:关键路径测试全绿 + 应用能跑 + 用户逐个审 diff。
  2. **学习**:补全 / 复核该功能的 ADR(决策理由 + trade-off + 面试要点);必要时
     生成源码精读笔记。
  3. **合并**:功能整体合并回 `main`。
  4. **收尾**:里程碑处**再生** `BACKEND_OVERVIEW.md`;更新 `CLAUDE.md` 的
     「当前进度」。
- **谁做**:验证 / 拍板在主窗口;独立复审、整理笔记可派 subagent。
- **完成标准**:功能已合并;DoD 全部满足;学习文档到位。

## 4. 文档架构

### 4.1 三层文档(状态沉淀进 repo,换窗口不丢)

**为什么**:每开一个新窗口,AI 对之前一切一无所知(它"失忆")。跨窗口续接**不靠
AI 重读全部代码**,靠沉淀在 repo 里的分层文档——新窗口自动读一部分,用户一句话指路。

| 层 | 文档 | 作用 | 谁读 |
|---|---|---|---|
| 常驻层(自动加载) | 项目根 `CLAUDE.md` | 长期规矩:流程、文档结构、约定 | 每个新窗口开机自动读 |
| 流程层 | `docs/WORKFLOW.md` | 每个 phase 具体怎么操作(proto-skill) | 需要时读 |
| 任务层 | `docs/changes/NNNN-*.md` | 某功能改到哪了、下一步、交接段 | 干这个功能时读 |

**贯穿原则:自动加载的文档必须有界(bounded)。** 常驻层(`CLAUDE.md`)每个窗口都加载,
越短越好——任何会随进度**无限增长**的内容(里程碑历史、完成流水)一律不进常驻文档:
「当前进度」这类只写"现在"的内容用**覆盖式更新**保持固定长度,历史交给 `git log` 和
`docs/changes/` 里的已完成工单。
> 注意区分:工单里的「变更记录」是**追加式**的,但那没问题——工单不自动加载、只在做该
> 功能时才读,且每份只覆盖一个功能,天然有界。判据是:**会不会每次都被加载 + 会不会无限增长。**

### 4.2 派生文档 vs 源头文档(治"文档漂移"的根本办法)

| | 派生文档(Derived) | 源头文档(Source-of-truth) |
|---|---|---|
| 例子 | 架构总览、Swagger API 文档 | ADR、`api-contract.md`、功能工单 |
| 本质 | 代码的"缓存",能从代码再生 | 含人类意图 / 决策,代码里生不出来 |
| 维护 | **不手动维护,里程碑再生** | **手动维护,列入 DoD** |

三条策略:
1. 接口文档尽量用 **Swagger / 注解自动生成**——改注解即改文档,永不漂移。
2. 架构总览当"快照",**每个 Phase / 大功能完成后再生一次**,中间小改动不碰它。
3. 把"更新文档"写进 **DoD**——功能不是"代码写完"算完,是"代码+测试+文档+ADR"
   都更新才算 `DONE`。

### 4.3 推荐目录结构(通用)

```
<project-root>/
├─ <后端代码目录>/
├─ <前端代码目录>/
├─ reference/                     # [只读] 参考资料(如被替换的旧客户端源码),禁改、不构建、不扫描
├─ CLAUDE.md                      # 协作宪法,自动加载
└─ docs/
   ├─ WORKFLOW.md                 # 5 阶段操作手册(本方法论的项目实例)
   ├─ BACKEND_OVERVIEW.md         # [派生] 架构总览,里程碑再生
   ├─ api-contract.md             # [源头] 前后端接口契约
   ├─ changes/                    # 每个功能一份工单
   │    └─ 0001-<slug>.md
   └─ decisions/                  # ADR,永久学习资产
        └─ 0001-<slug>.md
```

**关键**:所有文档**进 git,和代码一起提交**——checkout 到历史提交时,看到的是那时的
文档状态,防止文档和代码脱节。文件名用**数字前缀 + slug**(`0001-...`)保证顺序可读。

## 5. 关键概念与协议(工程实践词典)

- **ADR(Architecture Decision Record,架构决策记录)**:记录"我们当时为什么这么选"
  的标准格式。它既是工程规范,也是本方法论的**核心学习资产**——面试最爱问"为什么用 X
  不用 Y",ADR 写的正好是这个。模板见 9.3。

- **契约优先(Contract-First)**:全栈功能实现前先定死前后端接口契约。它是前后端的
  唯一事实来源,也是二者能并行的前提。

- **完成的定义(Definition of Done, DoD)**:一个功能"完成"= 代码 + 测试 + 相关文档
  + ADR 全部更新。文档更新是"完成"的一部分,不是事后补。

- **上下文交接协议(Context Handoff)**:长任务会超出单窗口上下文。每个窗口收工前,
  AI 必须更新工单里的「⭐ 交接段」:当前在哪、下一步做什么、**别碰哪些文件**、验证命令。
  新窗口读这一段 + 清单即可秒接,不必重读全部代码。

- **复述握手(Verification Handshake)**:新窗口 AI 读完状态后,**先向用户复述**"当前
  在哪步 / 下一步 / 要改哪些文件 / 怎么验证",用户确认后才动手。防止"理解偏了还一路
  狂奔"。

- **特征测试(Characterization Test)**:重构前,先写测试把"当前行为"钉死(哪怕当前有
  bug 也照实记录),之后改动只要测试还绿就证明外部行为没变。是"AI 改、我验收"能成立的
  物质基础。

- **只读参考区(`reference/`)**:被替换的旧客户端 / 第三方参考代码放这里,标记只读、
  排除构建、排除扫描、排除 AI 修改范围。用它之前**先榨取价值**(如从旧客户端提取出
  接口契约写进 `api-contract.md`),用完即可归档。

- **派生文档里程碑再生**:能从代码再生的文档(架构总览)不逐次手改,大节点重跑一次。

## 6. subagent 委派策略

**主窗口是用户的指挥席**:用户只跟它说话,它自己判断何时把活外包给 subagent。用户
不需要手动管理一堆 agent。

判断规则:subagent 上下文**隔离**(看不到对话,只返回结果)——
- ✅ **外包**:边界清晰、读取量大、只要结果不要过程的活。
- ❌ **自留**:需要和用户反复探讨、用户必须在环做决策的活。

| Phase | 主窗口(用户在环) | 外包给 subagent |
|---|---|---|
| 0 安全网 | git、跑起来 | 写关键路径测试 |
| 1 理解 | 看报告、提问 | **整个扫描(读海量文件)** |
| 2 规划 | **全程:探讨、写 ADR、定契约** | 顶多派 explore 查"某标识符在哪被用" |
| 3 执行 | 编排、审 diff、把测试门 | **每一步实现 = 一个 subagent** |
| 4 验证学习 | 跑测试、审 diff、拍板 | 独立 verifier 复审;整理 ADR |

要点:
- **不是"一 phase 一 subagent"**:Phase 1 可能一个大 subagent;Phase 2 一个都不外包;
  Phase 3 是**一步一个**。
- **同一份功能工单一物两用**:对新窗口主 AI 是"交接文档",对派出的 subagent 是
  "任务简报"。所以工单要写清"现状 / 背景 / 要改哪些文件"。
- subagent 还能**保护主窗口上下文**:让它读几十个文件、只返回浓缩结果,主窗口始终
  清爽——正好治"长任务窗口不够用"。
- **现阶段建议**:起步时能在主窗口做的就在主窗口做,只外包最明显的重活(扫描)。
  跑顺一个完整功能、有了手感,再逐步把 Phase 3 实现步骤也交给 subagent。

---

## 7. 分支与提交策略

- **一功能一分支**:`feature/<slug>`。
- **一步一 commit**:工单清单里每个步骤对应一个 commit;message 说清"改了什么 +
  关联工单/步骤"。
- **一功能一次合并**:整个功能验收通过后合并回 `main`。
- 好处:审 diff 颗粒度细,任何一步出错都能单独回滚。

---

## 8. 元原则:先手动跑,后固化成 skill

`docs/WORKFLOW.md` 是"活文档",有双重身份:

- **现在**:手动跑流程的操作手册。
- **将来**:某个 phase 跑顺、步骤稳定了,它对应那节的内容**就是那个 skill 的规格
  说明**,可交给 skill-creator 固化成 skill。

**所以"先手动跑"不是浪费,是在给未来的 skill 攒真实规格。**
> 类比:先自己走一遍这条路线,记下"哪个路口容易走错",这份笔记才是将来做"自动导航
> (skill)"的数据。没走过就做导航,一定脱离实际。

**推荐节奏**:先纯手动跑通第一个真实功能(当小白鼠),跑顺了再把 phase 逐个固化成
skill。别一上来就做工具。

### 9.1 项目 `CLAUDE.md` 模板(为每个新项目实例化)

```markdown

用户是 Tech Lead:定方向、审设计、做决策。带设计决策的事先摆方案→探讨→拍板→写 ADR。

7. 派生文档不手改,里程碑再生;接口文档尽量注解自动生成。
8. 外包劳动、自留决策。
9. reference/ 只读,禁改、不构建、不扫描。

## 五、文档地图 / 六、5 阶段流程
```

## 10. 用这份文档启动一个新项目(操作清单)

当用户下载了一个新的教学项目,想套用这套工作流时:

1. **把 GOOD.md 放进新项目根目录**(或把它交给新窗口的 AI 阅读)。
2. **AI 通读 GOOD.md**,理解方法论。
3. **AI 为新项目生成专属文档**:
   - 根据 9.1 模板生成项目 `CLAUDE.md`(填入该项目的技术栈、改造目标、平台特定点)。
   - 根据本方法论生成 `docs/WORKFLOW.md`(5 阶段操作手册的项目实例)。
   - 建好 `docs/changes/`、`docs/decisions/` 目录。
4. **AI 先做 2.1 的判断**:识别这个项目里"有价值的业务后端"和"平台特定噪音",和用户
   一起确定改造目标(先别删任何东西)。
5. **和用户确认后,进入 Phase 0**(安全网)。
6. 之后按 Phase 1→4 循环推进每个功能。

> 一句话:GOOD.md 是"种子",AI 读完它,为具体项目长出 CLAUDE.md + docs 结构,
> 然后就能按统一的工作流干活了。
