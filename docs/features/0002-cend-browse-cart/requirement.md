# [0002] C 端重建①:商品浏览 + 购物车 — Requirement

## 元信息
- 编号: 0002
- 类型: 新功能(C 端重建 epic 的第一块,见 `docs/blueprint.md`)
- 档位: T2(多步骤 / 前端跨模块;含 1 份轻 ADR:UI 方案 + 登录门槛)
- 状态: 已基线(2026-07-22)
- 关联: Proposal → ./proposal.md | Progress → ./progress.md | ADR → ../../decisions/0002-cend-browse-cart.md | 契约 → ../../api-contract/用户端接口.md

## 1. 背景与动机 (Why)
C 端 Web(`project-sky-user-vue3`)目前只有 0001 的认证闭环(登录/注册/改密/登出),**没有任何点餐业务**。要照 `reference/` 微信小程序把 C 端端到端重建,第一块必须是"顾客能浏览商品、把想吃的加进购物车"——这是「浏览 → 下单 → 支付」整条链路的**入口和地基**:后续 0003 下单直接读购物车,没有它后面无从谈起。同时这一块要把**整个业务前端的地基**打好(组件库、API 调用模式、登录门槛),供 0003~0005 复用。

## 2. 目标 (What)
做完之后:顾客登录后进入**点餐主页**,能按分类浏览菜品与套餐、查看菜品口味规格 / 套餐含菜、把菜品(可选口味)或套餐加入购物车、增减数量、清空;购物车条目与**商品合计金额实时正确**,且**刷新后仍在**(以服务端为准)。UI 用 Vant,最低可用即可。

## 3. 范围 (Scope)

### 做什么 (In Scope)
- **店铺营业状态**展示(`GET /user/shop/status`)
- **分类浏览**:左侧分类栏(`GET /user/category/list`,type 1 菜品 / 2 套餐)
- **菜品浏览**:按分类查菜品,数据含口味 `flavors`(`GET /user/dish/list`)
- **套餐浏览**:按分类查套餐 + 套餐含菜明细(`GET /user/setmeal/list`、`GET /user/setmeal/dish/{id}`)
- **菜品详情 / 口味规格选择**:照 reference,做成点餐页内**弹层**,不做独立页
- **购物车**:加 `add` / 减 `sub` / 查看 `list` / 清空 `clean`(全部走已有后端接口)
- **点餐主页布局(Vant)**:左分类 + 右商品列表 + 底部购物车栏(合计金额 + "去结算"按钮)
- **引入 Vant** 到 `project-sky-user-vue3`(见 ADR-0002)
- **整站登录门槛**:未登录访问点餐页 → 跳登录(复用 0001 路由守卫);登录/注册页复用 0001(见 ADR-0002)
- **前端资产**:新增 `api/`(shop/category/dish/setmeal/cart)+ 购物车 Pinia store(服务端为准,写操作后重拉)
- **商品图片**:菜品 / 套餐图片统一用**占位图 `Dummy.png`**(不接任何云存储);把它从只读的 `reference/Resource/Dummy.png` **复制**进前端资产目录再引用(reference 只读、不纳入构建,故不能直接引)。见 ADR-0002 D3
- **后端一处 bugfix**(评审发现):修 `ShoppingCartMapper.updateNumberById`——SQL 写成 `set amount`(应为 `set number`),导致购物车数量恒为 1、"增减数量"失效。此为本功能**唯一**后端改动。见 ADR-0002 Addendum AD1

### 不做什么 (Out of Scope)
- 地址簿、下单结算、备注 → **0003**("去结算"按钮的目标页属 0003;本功能里该按钮先占位 / 禁用 / 跳占位路由)
- 支付、支付成功页 → **0004**
- 订单历史 / 详情 / 催单 / 取消 / 再来一单 / 我的 → **0005**
- 后端**其它任何改动**(本功能只含上面那一处购物车 bugfix;若再发现别的后端问题,另开工单,不夹带)
- 配送费 / 餐盒费的结算口径(属下单,0003);本功能购物车只展示**商品合计**
- 接入**云存储读取真实商品图片**:**明确不用阿里云**(现在和以后都不用);将来升级走 **AWS S3**(用户目前对 AWS 尚不熟,列为 epic 外将来增强,见 blueprint)。本功能**所有商品图片一律走占位图**,不接任何云

## 4. 验收标准 (Acceptance Criteria)
> 可观测、面向行为、含负例。DoD(流程门)见 CLAUDE.md 铁律 4,此处不重抄。

- [x] 登录后进入点餐主页,顶部显示营业状态(营业 / 打烊);**Redis 未初始化 / `shop/status` 异常时页面不白屏**(状态栏显示"营业状态未知/加载失败",且不阻塞分类与菜品浏览)
- [x] 左侧显示分类列表;点击某分类,右侧切换为该分类的菜品或套餐列表
- [x] 菜品项显示名称 / 占位图 / 价格;有口味的菜品可打开**规格选择弹层**
- [x] 套餐项可查看**套餐含菜**明细(名称 / 份数 / 描述)
- [x] "加入购物车":无口味菜品直接加;有口味菜品**选完规格再加**;套餐可加
- [x] 底部购物车栏实时显示条目数与**商品合计金额**;可通过购物车明细(CartDetailPopup)对单品增 / 减数量、清空
- [x] **(数量硬断言 — 证伪核心功能,针对已知 bug)**:同一菜品连加 3 次 → 该行 `number` = 3、合计 = 单价×3;**刷新页面后仍为 3**(不是恒 1)。用 `cart/list` 返回值核对,不靠肉眼
- [x] 购物车所有写操作请求头带 `Authorization: Bearer`(复用 0001 request 层)
- [x] 打烊时:商品仍可浏览 / 加购,但"去结算"置灰并提示"店铺打烊"(0002 后端不拦打烊加购)
- [x] (负例)未登录直接访问 `/menu` → 守卫拦截跳登录;**已登录浏览中登出 → 再访问 `/menu` → 仍被拦截、无闪烁**;无 token 直打后端购物车接口 → 401
- [x] (负例)有口味的菜品未选规格时点"加" → **不发出 `cart/add` 请求**,并以 Toast/弹层提示"请选择规格"(照 reference 交互)
