# [0003] C 端重建②:地址簿 + 下单 — Requirement

## 元信息
- 编号: 0003
- 类型: 新功能(C 端重建 epic 的第二块,见 `docs/blueprint.md`)
- 档位: T3(全栈:前端多页 + 后端改动 + 契约校准;含 ADR 六决策)
- 状态: 规划中(Phase 2,2026-07-22)
- 关联: Proposal → ./proposal.md | Progress → ./progress.md | ADR → ../../decisions/0003-addressbook-order.md | 契约 → ../../api-contract/用户端接口.md

## 1. 背景与动机 (Why)
0002 已让顾客能浏览商品、把想吃的加进购物车(以服务端 `shopping_cart` 为准),但"去结算"仍是占位——购物车里的东西还送不出去。0003 补上「浏览 → **下单** → 支付」链路的中间关键一环:顾客得先有**收货地址**,才能在**结算页**确认订单并提交,生成一笔真实订单(`orders` + `order_detail`)。这是支付(0004)、订单管理(0005)的前置:没有下单,后面无从谈起。

同时 0003 要清理下单主链上两个后端历史包袱:①`submitOrder` 里同步调**百度地图**做配送范围校验(强依赖外部 API,且该段代码本身已坏 —— 见 ADR D2);②`submitOrder` 三步写库(建订单 / 建订单明细 / 清购物车)**没有事务**,中途失败会留下脏数据。二者都在 0003 的后端改动范围内。

## 2. 目标 (What)
做完之后,登录顾客能——
- **管理收货地址**:新增(含**三级省市区**选择)、编辑、删除、设默认、查看列表;
- 从购物车"去结算"进入**结算页**,看到默认地址 / 切换地址、购物车明细、金额(商品合计 + 打包费 + 配送费)、填备注、选餐具数量与送达时间;
- 点"去支付"提交订单 → 后端**原子化**生成订单(三写全成或全不成)、清空购物车、返回订单号 / 金额 / 下单时间 → 前端落到"订单已创建"占位页(支付由 0004 接管);
- 用**任意合法收货地址**都能下单(**不再被"超出配送范围"拦截**)。

UI 用 Vant,行为对齐 `reference/` 小程序,最低可用即可。

## 3. 范围 (Scope)

### 做什么 (In Scope)
- **地址簿前端**:列表页 / 新增-编辑页(**同一页**,reference 风格,`?id=` 区分编辑)/ 删除 / 设默认 / 查看,全走已就位的 `/user/addressBook/**` 7 个接口。
- **三级省市区选择**:`van-area` + `@vant/area-data`(见 ADR D1),纯客户端静态区划数据,**无区域 API**。
- **结算 / 确认订单页**:地址卡(默认地址预填 `GET /addressBook/default`,可跳地址簿选)、购物车明细(**复用 0002 cart store**)、金额展示、备注、餐具数量、送达时间(立即 / 选具体时间),提交走 `POST /user/order/submit`。
- **下单成功落地**:占位"订单已创建"页(展示订单号 / 金额),供 0004 用支付页替换(见 ADR D5)。
- **后端改动 1 —— 去百度配送校验**:删 `OrderServiceImpl.checkOutOfRange` 及其在 `submitOrder` 内的调用,清理相关 `sky.baidu.ak` / `sky.shop.address` 依赖(见 ADR D2)。
- **后端改动 2 —— submitOrder 补事务**:`@Transactional` 包住"建订单 → 建订单明细 → 清购物车"三写,任一失败整体回滚(见 ADR D3)。
- **后端改动 3(bugfix)—— 地址簿越权修复**:给 `AddressBookMapper` 的 `getById` / `update` / `deleteById` 加 `user_id` 归属校验,堵住 IDOR(见 ADR D6),**独立 commit**。
- **前端资产**:`api/address.ts` / `api/order.ts` + 业务类型 + 路由(结算页 / 地址簿页 / 下单成功页 + 0002"去结算"接线)+(如需)地址 store。
- **契约校准**:把 `docs/api-contract/用户端接口.md` 的 addressBook CRUD + order/submit 与后端实际对齐,补金额 / 打包费 / 配送费口径注释、"去百度校验后行为"注释(见 ADR D4)。

### 不做什么 (Out of Scope)
- **支付**:`PUT /user/order/payment`、支付页、成功页、mock 支付、去 openid → **0004**。0003 下单成功只落占位页,"去支付"不真正发起支付。
- **订单管理**:历史订单 / 详情 / 催单 / 取消 / 再来一单 / 用户中心 → **0005**。
- **金额后端重算 / 反篡改**:0003 照 reference 由前端算 `amount`、后端信任存库(见 ADR D4);服务端重算列为将来增强,ADR 记风险。
- **配送费真实计算**:照 reference **写死 ¥6**;真实配送计费 out of scope。
- **打烊拦截下单**:后端 submit 不新增营业状态校验(reference / 现后端本无此校验,保持行为一致);前端沿用 0002"打烊置灰去结算"。
- **地址簿接口新增 / 改签名**:7 个接口已就位,除越权 bugfix 外不改契约、不加端点(照 blueprint「改造复用」)。
- `reference/` 堂食扫码点餐:整个 epic out of scope。

## 4. 验收标准 (Acceptance Criteria)
> 可观测、面向行为、含负例;测试门做成**可证伪断言 + 工具核对**(curl / DB 查 / preview_network),不留肉眼判定空间(沿用 0002 风格)。DoD(流程门)见 CLAUDE.md 铁律 4,此处不重抄。

**A. 地址簿**
- [ ] **新增**:填收货人 / 手机号 / 性别 / 三级省市区 / 详情 / 标签 → 保存 → 列表出现该地址;`GET /addressBook/list` 返回值含刚存的省市区 **Name**(不只 Code)。
- [ ] **编辑**:进编辑页预填原值(`GET /addressBook/{id}`)→ 改详情 → 保存 → 列表 / 详情为新值。**编辑页不提交 `isDefault`**(设默认只走 `/default`)。
- [ ] **设默认(硬断言:同一时刻至多一个默认)**:对 A 设默认 → 再对 B 设默认 → `list` 中**仅 B** `isDefault==1`、A 归 0。
- [ ] **(硬断言 · 编辑不误改默认)** 把 A 设为默认 → 进 A 编辑页只改详情 → 保存 → `list` 中 A **仍 `isDefault==1`**(证编辑不吞默认;评审 AD1)。
- [ ] **删除** → 列表不再含该 id。
- [ ] (负例)手机号非 11 位数字 / 收货人为空 → 前端拦截**不发请求** + 提示。
- [ ] **(负例 · 越权 · 硬断言)** 前置:账号甲有地址 `id=X`、**账号乙无 `id=X`**(先查断言)。用乙 token 直打 `GET/PUT/DELETE /user/addressBook`(id=X)→ **拿不到 / 改不动 / 删不掉**(返回空 / 错误,不泄露 / 不篡改甲的数据)。**再补一发**:乙 token + `PUT` body `{id:X, userId:甲Id}`(伪造 userId)→ **甲地址仍不变**(证归属只认 `BaseContext`、不认 body;评审 AD1)。curl 硬验。

**B. 结算 + 下单**
- [ ] 购物车有货时"去结算"进结算页;页面显示默认地址(无默认时提示"请选择收货地址")、购物车明细(**复用 0002 store**,条目 / 金额与购物车一致)、商品合计 + 打包费 + 配送费 ¥6。
- [ ] **换地址**:跳地址簿选一个 → 回结算页地址卡更新为所选。
- [ ] 可填备注、选餐具数量、选送达时间(立即 / 具体时间)。
- [ ] 提交时 `deliveryStatus` / `tablewareStatus` **必传非空**(前端定死默认 `deliveryStatus=1` / `tablewareStatus=1` 随请求发出);它们是 `orders` 的 `NOT NULL` 列,漏传显式 NULL 会 submit 500。测试门:字段齐全提交成功 + curl 故意省略这两字段 → 500(证前端定死是必要的;评审 AD1)。
- [ ] 提交订单 → 返回订单号 / 金额 / 下单时间 → 落"订单已创建"占位页并显示订单号;**下单成功后购物车被清空**(`shoppingCart/list` 为空)。
- [ ] **(硬断言 · 下单原子性)** 在三写**全部之后(清购物车之后)**注入 `RuntimeException` → `orders` 表**无该笔残留** 且 **购物车仍在**(证 `@Transactional` 回滚含清购物车这步)。**注入点必须在清购物车之后**,否则"购物车仍在"恒真、无法证伪(评审 AD1:清购物车是三写末步)。对照:去掉 `@Transactional` 应看到脏订单 + 购物车已清。用 DB 查 + `shoppingCart/list` 核对。
- [ ] **(硬断言 · 去百度校验)** 用任意 / 远距离收货地址下单 → **成功**,不再返回"超出配送范围"(修复前该路径会因假 AK / 百度失败而 500 或被拒)。
- [ ] **(负例 · amount 防呆)** 构造 `amount<=0`(0 或负数)的 `order/submit` → 后端拒(`OrderBusinessException`),不生成订单。curl 硬验。
- [ ] **(负例 · 越权 · 下单读地址)** 用乙 token 提交 `order/submit`、body `addressBookId` 填**甲的地址 id** → 后端拒(归属校验失败 / `ADDRESS_BOOK_IS_NULL`),不生成订单(证 submit 读地址也做归属校验)。
- [ ] (负例)未选地址点"去支付" → **不发 `order/submit` 请求** + 提示"请选择收货地址"(照 reference)。
- [ ] (负例)购物车为空提交 → 后端 `SHOPPING_CART_IS_NULL`;前端结算页在购物车空时给出态、不允许进到提交。
