## 背景

0002 是「C 端完整重建」epic 的**第一块**(见 blueprint)。除了实现浏览 + 购物车本身,它还要给**整个业务前端(0002~0005 共用)**定三件地基:①用什么 UI 组件库、②怎么控制访问门槛、③商品图片从哪来。功能级动机见 Requirement §1,不重抄。

### 约束这些决策的关键事实
- **reference 是移动端外卖小程序**:目标体验是手机点餐(左分类 / 右列表 / 底部购物车栏 / 规格弹层)→ 影响 D1。
- **现有脚手架无任何 UI 库**:0001 只手写了 `.card` 等最小样式 → D1 的起点。
- **后端 `/user/**` 默认全需 token**:安全白名单只有 `register` / `login` / `shop/status`;`category` / `dish` / `setmeal` / `shoppingCart` 均需登录态(0001 的 `SecurityConfig`)→ 直接决定 D2。
- **reference 原本就"进点餐页先强制登录"**:目标是"业务行为与 reference 完全一致" → 支撑 D2。
- **不接阿里云(用户明确:现在与将来都不用)**;`dish.image` 原是阿里云 OSS URL,凭据失效会图裂 → 逼出 D3。
- **学习项目、UI 只保最低标准**:贯穿三个决策的裁剪标尺。

## 决策概览

| 编号 | 决策点 | 结论 |
|---|---|---|
| D1 | UI 组件库 | **引入 Vant**(移动端组件库) |

## D1 — UI 组件库:引入 Vant

### 方案对比
| 方案 | 优点 | 缺点 |
|---|---|---|
| **Vant(选)** | 移动端组件库,**数量步进器 / 弹层 / 卡片 / 底部栏 / 下拉刷新开箱即用**,最贴近小程序原体验;**恰恰最省 CSS**;Vue3 官方支持、按需引入(tree-shaking) | 多一个依赖;偏移动端(桌面场景不合,但本项目正是移动外卖) |
| 纯手写最小 CSS(维持现状) | 零依赖、与 0001 一致 | 步进器 / 弹层 / 结算栏都得手搓,**更费 CSS**——与"别花力气在 CSS"的目标背道而驰 |
| Element Plus | 组件全、admin 端生态成熟 | **桌面向**,移动外卖场景不搭 |

### 决策
选 **Vant**。核心理由:目标是移动外卖体验且**要少写样式**,Vant 直接提供点餐页要用的全部交互组件,反而比手写更省事;Vue3 一等支持。这不是"因为不会写 CSS",而是"把有限精力放在前后端交互正确性、不放在重造轮子上"(符合 CLAUDE.md 学习目标)。

---

## D2 — 访问门槛:整站登录后可用(不改后端白名单)

### 关键技术事实
后端 `/user/**` 除 `register`/`login`/`shop/status` 外**全部要 token**。所以"能不能匿名浏览菜品"不是纯前端问题——`GET /user/dish/list` 等本身就会对无 token 请求返回 401。真正可选的两条路:

| 方案 | 优点 | 缺点 |
|---|---|---|
| **整站登录后可用(选)** | **行为与 reference 完全一致**(reference 进点餐页即强制登录);**后端 `SecurityConfig` 零改动**;复用 0001 路由守卫;最小可用 | 不能匿名逛(但 reference 本就不能) |
| 匿名浏览、下单才登录(更像真电商) | 更贴近北美电商习惯(公开商品目录) | 要改后端白名单(放开 `category`/`dish`/`setmeal` 的 GET);偏离"与 reference 一致";本功能范围变大 |

### 决策
选 **整站登录门槛**。理由:目标是"业务行为与 reference 完全一致",该方案直接达成且**不动后端**(0002 保持"后端零改动"边界);前端复用 0001 已有的路由守卫(`beforeEach` 未登录跳 `/login`)。匿名浏览留作**将来增强**记录在案。

> 注意:前端路由守卫只是 **UX 层**兜底(拦一下、给跳转体验),**真门槛在后端**——无 token 打 `/user/**` 一律 401。这条 authn/authz 边界与 0001 一脉相承。

## D3 — 商品图片来源:占位图,不接云存储

### 方案对比
| 方案 | 优点 | 缺点 |
|---|---|---|
| **占位图 `Dummy.png`(选)** | 不依赖任何云凭据、**前端开发与存储彻底解耦**、一定能显示;把媒体来源做成"一个可替换的点" | 看不到真实菜品图(学习项目无所谓) |
| 阿里云 OSS(现状,`dish.image` 原值) | 数据库里就是这些 URL,零改 | **用户明确不用阿里云**;凭据失效则图裂;绑死特定厂商 |
| 直接接 AWS S3 | 目标终态、贴近北美栈 | 用户目前不熟 AWS;超出当前时间预算 |

### 决策
当前一律用 **占位图 `Dummy.png`**(不接任何云)。将来升级迁 **AWS S3**(epic 外 backlog,待用户熟悉 AWS)。**约束**:`reference/` 只读、不纳入构建,故执行期把 `reference/Resource/Dummy.png` **复制**进前端资产目录再引用,不直接引 reference。把取图封装成 `ProductImage`,其 prop 契约设计成**支持动态 URL**(当前恒传占位图),将来接 S3 时"仅当 image 空/加载失败才回退占位图",调用方不用改。

## D4 — 购物车同步策略:写操作后全量重拉(服务端为准)
> 评审(内 / 外审)补充的决策点,记录在此。

### 方案对比
| 方案 | 优点 | 缺点 |
|---|---|---|
| **写后全量重拉 `cart/list`(选)** | 实现最简、永远与后端一致、天然抗前端脏数据;契合"服务端为准" | 每次写多一次 GET;弱网下短暂延迟 |
| 乐观本地更新(先改 UI 再发请求) | 交互跟手、请求少 | 要处理失败回滚 / 版本冲突;后端无版本字段;易与"服务端为准"打架 |
| 差量更新(用 add/sub 响应体增量改) | 请求少 | 后端 add/sub 只回字符串、无变化条目,拿不到增量 |

### 决策
选**写后全量重拉**。学习项目并发低、后端无版本字段,重拉最简且最稳。**注意竞态**:连点加/减会并发多请求,后返回的可能比先返回的旧 → cart store 层对连续写做**串行化 / 忽略在途**(不靠 request 层),避免购物车数字跳变。

## Trade-off / 后果
> 要改哪些文件、怎么改见 proposal.md;这里只记决策层面的后果。

- **换来**:一套贴合移动外卖的 UI 基座(Vant)+ 与 reference 一致且零后端改动的访问模型 + 与云存储解耦的图片方案——三者都为 0003~0005 复用打底。
- **放弃 / 代价**:多一个前端依赖(Vant);不能匿名浏览;看不到真实菜品图。
- **后续义务 / 遗留**:
  - 匿名浏览 → 将来增强(需放开后端白名单)。
  - 真实图片 → 将来接 **AWS S3**(epic 外 backlog,见 blueprint)。
  - Vant 的引入方式(全量 vs 按需)在 proposal 里定实现细节。

## 💡 面试要点(广度卡片)
- **UI 组件库选型**:按**平台契合度**(移动 → Vant / 桌面 → Element)、维护活跃度、按需引入(bundle size)权衡;"要不要重造轮子"——把精力放在业务正确性而非重写通用组件。
- **前端路由守卫 vs 后端鉴权**:守卫是 **UX 兜底**(体验),真门槛永远在后端(无 token → 401)。"前端能不能保证安全"是经典陷阱题,答案是不能,前端可绕过。
- **公开目录 vs 登录墙**:真电商商品目录通常**匿名可看**(SEO / 转化),购物车 / 下单才要登录;本项目为对齐 reference 选了全站登录墙——能讲清"为什么真产品通常不这么做"就是加分。
- **媒体资源解耦**:图片来源做成"一个可替换的点"(占位图 → 对象存储 → CDN + 签名 URL),避免业务代码绑死某厂商;迁移只换取图策略。
  - 值得深挖 → 建议将来开 `divedeep`:对象存储 / 签名 URL / CDN 的取图链路(接 AWS S3 时)。

**① 两路收敛(高置信,已改进计划):**
