## 认证约定(0001 改造)

> 本次改造(微信登录 → 本地账号密码 + JWT + Spring Security 统管全站)统一了 C 端与管理端的认证方式。以下约定对所有需登录的接口生效。

- **认证头**:注册 / 登录成功后返回 `token`(JWT)。此后所有需登录接口,请求头带
  `Authorization: Bearer <token>`(替代旧版 C 端自定义头 `authentication`)。
- **JWT 载荷**:`{ sub: 用户id, role: USER, exp: 过期时间 }`;管理端为 `role: ADMIN`。
- **免认证接口**:`/user/user/register`、`/user/user/login`、`/user/shop/status` 无需 token;其余 `/user/**` 均需有效 token。
- **统一返回结构**:所有接口返回 `Result`,即 `{ code, data, msg }`(`code` = 1 表示成功)。
- **鉴权失败**:未携带 / 无效 token → `401 Unauthorized`;已登录但无权限 → `403 Forbidden`。

## 口味(dishFlavor)约定(0002 实测补注)
- 菜品 `flavors[].value` 是 **JSON 数组字符串**,如 `["不辣","微辣","中辣","特辣"]`;前端需 `JSON.parse` 出可选项。
- 加购 / 减购的 `dishFlavor` 是前端选定后的字符串,后端**原样存储**、`shoppingCart/list` **原样返回**;后端按 `dish_flavor` **精确匹配**判断"是否同一商品行"。→ 前端对某行做 +/-,**必须回传 `list` 返回的 `dishFlavor` 原值**,不可前端重新拼接。

## 依赖 Redis 的读接口(0002 实测补注)
- `GET /user/dish/list`(手写 `RedisTemplate` 缓存)、`GET /user/setmeal/list`(`@Cacheable`)、`GET /user/shop/status` 都读 Redis;**Redis 未起时它们返回 500(不降级)**。前端 `shop/status` 有兜底(取不到显示"未知"),但 dish/setmeal 无兜底——启动前须 `docker start sky-redis` + `PUT /admin/shop/1` 初始化。

## 地址簿 + 下单约定(0003 校准补注)
> 0003「地址簿 + 下单」对下列接口做了源码校准 + 行为变更。校准依据:后端 `AddressBookController` / `AddressBookServiceImpl` / `AddressBook` 实体 + `OrderServiceImpl.submitOrder` + `OrdersSubmitDTO` / `OrderSubmitVO` 实读;决策见 `docs/decisions/0003-addressbook-order.md`。契约签名(路径/方法/字段)不变,本节只补口径与行为。

**认证与当前用户**
- `/user/addressBook/**` 与 `/user/order/**` 均需 `Authorization: Bearer <token>`(除文件顶部列的三个免认证接口)。
- 当前用户由后端 `BaseContext`(ThreadLocal,JWT 拦截器注入)取,前端不必传 `userId`。
  - `POST /addressBook`(新增):后端 `save` 强制注入 `userId = 当前用户`,body 里的 `userId` 无效。
  - **`PUT` / `DELETE /addressBook`(D6 归属)**:归属**由后端按当前登录用户判定、绝不认 body 里的 `userId`**;即"userId 只认 `BaseContext`"。(注:此处**纠正**了早期"userId 一律忽略"的说法 —— 对 update/delete 而言 userId 是承载安全语义的字段,但取值来源是服务端而非请求体。)

**地址簿字段口径(与实体一致)**
- `sex` 是 **String**("0" / "1");`isDefault` 是 **Integer**(0 否 / 1 是);`label`(标签)为字符串,reference 用 1=公司 / 2=家 / 3=学校(前端展示映射)。
- 省市区六字段全为 **String**:`provinceCode/provinceName`、`cityCode/cityName`、`districtCode/districtName`。0003 前端用 `van-area` + `@vant/area-data` 生成(ADR D1)。
- **`DELETE /user/addressBook`** 的 `id` 是 **query 参数**(`?id=101`),非路径参数。
- **`GET /user/addressBook/default`** 与 **`GET /user/addressBook/{id}`**:`default` 是字面路径段,与 `{id}` 变量并存,Spring 按字面优先匹配。
- **设默认**(`PUT /user/addressBook/default`,body `{id}`):后端先把该用户**所有**地址 `is_default` 置 0,再把目标置 1 → **同一用户同一时刻至多一个默认**。

**地址簿归属校验(0003 行为变更 · D6 越权修复)**
- 修复前:`GET /{id}` / `PUT` / `DELETE` **只按 `id`**、无 `user_id` 归属校验 → 越权(IDOR)。
- 修复后:三者查询/更新条件**带上当前登录用户 id**;跨用户访问返回空 / 不生效(不泄露、不篡改他人地址)。签名不变,仅行为收紧。

**下单 `POST /user/order/submit` 口径(校准 + 行为变更)**
- **请求体不带商品明细**:后端按**服务端购物车**(当前用户 `shopping_cart`)+ `addressBookId` 派生 `order_detail`;前端只送既有契约里的 DTO 字段,不送 item 数组。
- **必传非空字段**:`deliveryStatus`、`tablewareStatus`、`payMethod` 对应 `orders` 的 `NOT NULL` 列(`payMethod` 后端还是原始 `int`),**前端必须给定值**(前端定死默认:`deliveryStatus=1` 立即送出、`tablewareStatus=1` 按餐量、`payMethod=1`)。漏传 `deliveryStatus`/`tablewareStatus` → 显式插 NULL 进 NOT NULL 列 → **submit 500**(NOT NULL 的 DEFAULT 只在列被省略时生效,显式 NULL 不救)。`estimatedDeliveryTime`/`packAmount` 可空,"立即送出"传 `estimatedDeliveryTime` 当前时刻即可。
- **金额 `amount` 由前端计算并提交、后端信任存库**(照 reference,**不重算**):`amount = Σ(单价×数量) + 配送费 6 + packAmount`;`packAmount = 菜品总件数`;`deliveryStatus`:1 立即送出 / 0 选具体时间(配 `estimatedDeliveryTime`)。⚠️ **price-integrity 风险**:前端可篡改金额,生产必须服务端重算;本项目为对齐 reference 暂信任前端(ADR D4)。**唯一后端兜底**:`amount<=0` 拒(`OrderBusinessException`),挡负数/零金额脏订单(非重算)。
- **去百度配送校验(D2)**:0003 起 `submitOrder` **不再做配送范围校验** → **任意合法地址都能下单**,不再返回"超出配送范围"/"店铺地址解析失败"/"配送线路规划失败"(改前该校验强依赖百度地图 API + 假 AK,fresh 环境跑不通)。
- **原子性(D3)**:`submitOrder` 加 `@Transactional`,三写(建订单 → 建订单明细 → 清购物车)**全成或全不成**;失败整体回滚(默认对 RuntimeException 回滚)。
- **起始校验仍在 + D6 下单读地址归属**:地址不存在 → `ADDRESS_BOOK_IS_NULL`;**`addressBookId` 不属于当前用户 → 同样拒**(不能拿别人的 `addressBookId` 下单);购物车空 → `SHOPPING_CART_IS_NULL`;**无营业状态校验**(保持行为一致)。
- 响应 `OrderSubmitVO`:`{ id, orderNumber, orderAmount, orderTime }`。0003 下单成功后前端落"订单已创建"占位页;**支付 `PUT /user/order/payment` 属 0004,0003 不实现**。

## 订单管理约定(0005 校准补注)
> 0005「订单管理」对下列既有端点做**源码校准 + 归属行为变更**(签名不变)。校准依据:`OrderServiceImpl`(`pageQueryForUser`/`details`/`userCancelById`/`repetition`/`reminder`)+ 管理端 `rejection`/`cancel` 实读;决策见 `docs/decisions/0005-order-manage.md`。5 端点均需 `Authorization: Bearer <token>`,当前用户由后端 `BaseContext`(ThreadLocal)取,前端不传 `userId`。

**订单归属校验(0005 行为变更 · D1 越权修复 · 仿 0003 D6)**
- 修复前:`orderDetail/{id}`(详情)、`cancel/{id}`(取消)、`repetition/{id}`(再来一单)、`reminder/{id}`(催单)**只按 `id` 取数、无 `user_id` 归属校验** → 越权(IDOR):可看/取消他人订单、把他人订单菜品灌进自己购物车、对他人订单催单。
- 修复后:这 4 端点在 Service 层**先取单并校验归属当前登录用户**,非本人 → 按 `订单不存在`(`ORDER_NOT_FOUND`)拒(不区分"不存在/非本人",避免存在性泄露);`repetition` 先取单校验归属再复制明细。**签名不变,仅行为收紧。** `historyOrders`(历史订单)本就按当前用户过滤,不变。

**历史订单查询 `GET /user/order/historyOrders`(口径校准)**
- Query:**`pageNum`(必)**、`pageSize`(必)、`status`(可空)。⚠️ **订正(0005 · AD1 HIGH#2)**:后端 `OrderController.page(int pageNum,int pageSize,Integer status)` 无 `@RequestParam`、靠**参数名绑定**,故分页页码参数名是 **`pageNum`** 而非原文档(本文件下方历史订单条目)写的 `page`——**用 `page` 发参会请求失败、非 `Result`**(Phase 4 实测 **HTTP 500** 而非原判的 400:`pageNum` 缺失后 `PageHelper.startPage` 收到 null 触发 NPE、被兜底成 500;不是 `MissingServletRequestParameterException` 400)。无论 400/500,前端**必须发 `pageNum`**。`pageNum`/`pageSize` 是原始 `int`=必填。
- **`status` 空 = 不过滤(全部)**;传值按订单状态过滤(1待付款/2待接单/3已接单/4派送中/5已完成/6已取消)。
- 响应 `PageResult`:`{ total: long, records: OrderVO[] }`(实体 `com.sky.result.PageResult`,字段名就是 `total`/`records`);每条 `OrderVO` 含订单主体字段 + **`orderDetailList`**(菜品明细:`name`/`image`/`number`/`amount`/`dishFlavor` 等,供列表卡片缩略)。
- **前端 tab→status 映射(0005 · 对齐 reference,3 tab)**:全部 → 不传 status;待付款 → `status=1`;已取消 → `status=6`。分页为 **触底无限加载**(`pageSize=10`,`records` 累加,`len>=total` 停止)。更细状态筛选属将来增强(端点已支持任意 status)。

**订单详情 `GET /user/order/orderDetail/{id}`(口径校准)**
- 响应 `OrderVO`:订单全字段(地址 `address`/收货人 `consignee`/电话 `phone`/订单号 `number`/`orderTime`/`amount`/`status`/`payStatus`/`payMethod`/`remark`/`packAmount`/`estimatedDeliveryTime` 等)+ `orderDetailList`(菜品明细)。**归属校验见上**。

**取消订单 `PUT /user/order/cancel/{id}`(口径校准 + 退款状态)**
- 仅 `status ≤ 2`(1待付款 / 2待接单)可由用户取消;`status > 2` → `订单状态错误`(`ORDER_STATUS_ERROR`)。成功后订单 `status=6 已取消` + `cancelReason="用户取消"` + `cancelTime`。
- **退款状态(D2)**:取消**待接单(status2 = 已支付)** 订单时,`pay_status` 置 `2 退款`(mock 退款仅 log,无真实资金流);未支付订单不置退款。**归属校验见上。**

**再来一单 `POST /user/order/repetition/{id}`(口径校准)**
- 把该订单的菜品明细**追加**进**当前用户**购物车(`shopping_cart`,后端 `insertBatch`,不清空原购物车)。**归属校验见上**(改前可复制他人订单)。**前端行为(0005 · AD1 Q1 拍板改合并)**:直接 repetition(**合并加入,不清空购物车**)→ 刷新购物车 → 回菜单页。(原"先清空再 repetition"的清空-重灌会在 repetition 失败时永久丢失服务端购物车,改为合并加入。)

**催单 `GET /user/order/reminder/{id}`(口径校准 + 状态守卫)**
- 通过 WebSocket 向管理端推"来单/催单提醒";成功返回 `code:1`(无 data)。**归属校验见上**(改前可对他人订单催单)。
- **状态守卫(0005 · AD1 Q3 新增)**:仅 `status==2 待接单` 可催单;非 status2(待付款/已接单/派送中/已完成/已取消)→ `订单状态错误`(`ORDER_STATUS_ERROR`)。改前无任何状态校验,可对任意状态订单催单。

**退款状态口径统一(0005 · D2 · 后端 · 含管理端)**
- 三处"已支付订单被取消要退款"统一为置 `pay_status=2 退款`:用户取消 `userCancelById`(既有,不变)、管理端拒单 `PUT /admin/order/rejection`、管理端取消 `PUT /admin/order/cancel`(后两者**改前只 log 不置 REFUND**,0005 补齐)。已支付判断三处统一用 **`Orders.PAID.equals(payStatus)`**(非 `==`,避免 Integer 装箱引用比较;AD1 订正)。**只补退款状态,其余状态流转/前置校验不动。**⚠️ **退款为 mock**:仅置状态位,无真实退款金额/时间/流水(生产需接真实退款,见 ADR-0005 后续义务)。

> **0004 校准(mock 支付,2026-07-23)**:此接口在 0004 从"微信 JSAPI 支付"改为 **mock 内部同步支付**——`payment()` 校验订单(存在 + 归属当前登录用户 + 未支付)后**直接内部置「已支付 / 待接单」并触发来单提醒**,不再调微信、不再依赖 openid、不再有异步回调(`/notify/paySuccess` 已删)。
> - **响应订正**:原文档写 `data.estimatedDeliveryTime`、旧代码实际返回微信预支付 5 字段(`nonceStr`/`paySign`/`timeStamp`/`signType`/`packageStr`)——**文档与实现本就漂移**。mock 后统一为「成功即可」:`data` 恒 `null`,前端凭 `code===1` 跳支付成功页;`OrderPaymentVO` 随之删除。
> - **幂等 / 归属**:对**已支付**订单重复调用 → 拒(`OrderBusinessException`"该订单已支付",弱防呆);`orderNumber` 属**他人**订单 → 按当前用户查不到 → 拒(归属校验)。
> - 详见 ADR-0004 D1/D2/D3。
